# 📊 TODO API 실측 벤치마크 결과 리포트

## 1. 개요 (Overview)
본 벤치마크는 다양한 백엔드 프레임워크를 기반으로 구성된 TODO API의 실측 데이터를 제공합니다. `make run` 워크플로우를 통하여 Docker 컨테이너 호스팅 환경에서 측정되었습니다.

- **실행 일시**: 2026년 3월 24일 10:45 KST
- **실행 환경**: macOS 15.7.4 (Apple M4 Pro) / Docker Desktop 기반
- **벤치마크 대상 및 버전**:
  - `go-gin`: Gin (권장 버전 1.10.0 기반)
  - `node-express`: Express (권장 버전 4.21.2 기반)
  - `java-spring`: Spring Boot (권장 버전 3.4.3 기반, JDK 21)
  - `python-fastapi`: FastAPI (권장 버전 0.115.8 기반, Python 3.13)
  - `php-laravel`: Laravel 11 (PHP 8.3)

---

각 서비스에 대한 500회 연속 순차 `curl` 요청을 바탕으로 수집된 결괏값입니다. (이번 측정치 기준, RPS 내림차순으로 정렬됨)

| :--- | :--- | :--- | :--- | :--- |

| **node-express** | 151.42 | 0.02% | 26.75 MiB | 239 MB |
| **java-spring** | 139.90 | 0.39% | 152.60 MiB | 324 MB |

## 3. 상세 분석 (Detailed Analysis)

### 🥇 절대적 리소스 가성비: Go (Gin)
- **분석**: Go는 9.27MiB라는 초경량 메모리 점유율과 164.26이라는 가장 높은 상대적 RPS를 기록했습니다. 동작 중 CPU 점유율은 0%대에 수렴하여 프레임워크 오버헤드가 사실상 전무함을 보여줍니다.
- **특이사항**: 컴파일 언어 특성의 쾌조를 띄며, 이미지 크기(56MB) 또한 타 언어 대비 현격하게 적어 마이크로서비스의 민첩성을 극대화합니다.

### 🥈 가벼운 고효율: Node.js (Express)
- **분석**: RPS 151.42를 기록하며 2위 구간을 안정적으로 지켰습니다. 사용된 메모리(26.75MiB) 역시 Go 바로 다음으로 적게 소모합니다.
- **특이사항**: 싱글 스레드 이벤트 루프에 기반한 인터프리터의 뛰어난 반응성을 여실히 보여줍니다.

### 🥉 엔터프라이즈의 중추: Java (Spring Boot)
- **분석**: RPS는 139.90 수준으로 측정되었습니다. JVM 특성상 메모리 점유율이 152.6MiB로 가장 높고 시작 속도의 오버헤드가 있습니다.
- **특이사항**: 이 벤치마크가 동시 접속자 수천 명을 흉내내는 Load Test가 아님을 고려하면, 웜업(Warm-up) 이후 처리량은 역시 대규모 운영 측면에서 손색이 없음을 나타냅니다.

### 🐍 데이터 및 비동기 개발의 핵심: Python (FastAPI)
- **분석**: RPS 139.06으로 Java와 동률 수준을 보였으며, 메모리 사용량은 47.87MiB를 기록했습니다. 이미지 사이즈(346MB)는 가장 큽니다.
- **특이사항**: Python 특성상 구동 이미지 크기나 시스템 비용은 크지만, 데이터 분석 모듈, 현대적인 비동기 개발 생산성 측면에서 강력한 포지션을 가지고 있습니다.

### 🐘 풍부한 기능으로 무장: PHP (Laravel)
- **분석**: 타 언어와 달리 유일하게 백엔드 풀 프레임워크 기반이기에, RPS는 114.71로 가장 낮고 5.84%라는 높은 상대적 CPU 점유율을 기록했습니다.
- **특이사항**: 순수 성능 수치만 보면 제일 뒤처지지만, Eloquent ORM이나 내장 로깅, 백엔드 관리 도구 등 방대한 기능을 고려한 훌륭한 "트레이드 오프" 선택지입니다.

---

## 4. 실험 한계 (Limitations)
- **동시성 처리능력 판단의 부재**: 본 벤치마크는 단일 `curl` 요청을 500회 연속 반복하는 스크립트로 구성되었습니다. 따라서 실무에서 필연적인 **동시 다발적 부하(Concurrent Load)**나 Thread/Async 풀 고갈 상황에서의 성능 방어 능력을 평가할 수 없습니다.
- **단일 로컬 하드웨어 제약**: M4 Pro 환경의 Docker Desktop 로컬 가상머신 위에서 측정되었으므로, Native Linux 서버 환경에서의 커널 레벨 네트워크 최적화나 퍼포먼스 양상과는 차이가 있을 수 있습니다.
- **단비즈니스 로직**: 현재의 테스트 컨트롤러는 초보적인 리턴 값만 가지므로, 실제 DB I/O, 복잡한 직렬화나 메모리 캐싱, 미들웨어 스케일 아웃이 발생할 때의 ORM 오버헤드를 반영하지 않습니다.
